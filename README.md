# Campaign & SEO Management Platform
## Full Project Design-First Blueprint

> **How to use this document.** This is the single source of truth for the entire project. Read it before writing any code for a given module. Every API contract, database decision, and architectural choice is explained with its reasoning — not just its "what" but its "why" — so you can defend every decision in an interview or onboarding conversation. Implementation comes *after* the design is understood.

## Part 0 — Architecture Overview

### The layered architecture (applies to every module)

Every feature in this project follows the same four-layer pattern: the **Controller** handles HTTP (parsing requests, setting status codes), the **Service** owns all business logic (ownership checks, calculations, orchestration), the **Repository** talks to the database, and the **Entity** represents a database row as a Java object. No layer skips another — a Controller never calls a Repository directly, and a Service never touches HTTP concepts like `HttpServletRequest`.

The **build order** within each module is always bottom-up: Entity → Repository → DTOs → Exception classes → Service → Controller → Tests.

### The domain hierarchy

```
User ──< Site ──< Campaign
              └──< Event
              └──< SeoReport ──< SeoCheck
                           └──< ReportSendLog
User ──< ReportSendLog
```

**`Site`** is the primary unit of organization — it represents a website where the tracking snippet is installed. Every campaign, event, and SEO report is scoped to a site, not directly to a user. The `User` owns `Site`s; everything else hangs off a `Site`.

This hierarchy has one important implication for security: ownership checks always go through `Site`. When a user requests data for a given `siteCode`, the service verifies ownership once via `siteRepository.findBySiteCodeAndUser(siteCode, user)` — one check that covers every downstream resource.

### Security model (applies to every endpoint)

All endpoints except `/api/auth/register` and `/api/auth/login` require a valid JWT in the `Authorization: Bearer <token>` header. The authenticated user is extracted from `SecurityContextHolder` using `@AuthenticationPrincipal`. **Clients never send their own `userId` in a request body** — they prove identity via the token only.

### Error response contract (applies to every module)

Every error returns the same JSON shape:

```json
{
  "status": 404,
  "error": "Not Found",
  "message": "Site not found"
}
```

This contract is enforced by a single `GlobalExceptionHandler` (`@ControllerAdvice`).

### Database conventions (applies to every table)

Every table uses `BIGINT GENERATED BY DEFAULT AS IDENTITY` as its primary key. Every enum column is stored as `VARCHAR` with `EnumType.STRING`. All timestamps are `TIMESTAMP WITHOUT TIME ZONE`. Foreign key columns follow the naming pattern `<table_singular>_id`.

## Part 1 — Auth Module

### API Contract

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/auth/register` | No | Create account, return JWT |
| `POST` | `/api/auth/login` | No | Validate credentials, return JWT |

**Request body (both endpoints):**
```
email       String   required, valid email format
password    String   required, 8–100 characters
```

**Response (201 Created for register, 200 OK for login):**
```
token       String   signed JWT, valid for 24 hours
```

Both endpoints return the same `"Invalid email or password"` message for wrong email *and* wrong password — this prevents user enumeration attacks.

### Database Schema

```sql
CREATE TABLE users (
    id            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email         VARCHAR(255)  NOT NULL UNIQUE,
    password_hash VARCHAR(255)  NOT NULL,
    created_at    TIMESTAMP WITHOUT TIME ZONE
);
```

### Key Design Decisions

BCrypt is used for password hashing because it is intentionally slow, generates a random salt on every hash, and has an adjustable cost factor. The column is named `password_hash` — not `password` — to make it impossible to accidentally treat it as a raw credential. There is no `role` column — a single-role system keeps auth simple. Spring Security's `getAuthorities()` returns an empty list, meaning all authenticated users have equal access.

## Part 1b — Site Module

`Site` is the entity that anchors every other resource. It must be created before any campaigns, events, or SEO reports can exist.

### API Contract

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/sites` | Yes | Create a site |
| `GET` | `/api/sites` | Yes | List all sites owned by the user |
| `GET` | `/api/sites/{site_code}` | Yes | Get a single site |
| `DELETE` | `/api/sites/{site_code}` | Yes | Delete a site |

**`POST /api/sites` — request body:**
```
siteName    String   required, not blank
```

**`SiteResponse` — returned by all endpoints:**
```
id          Long
siteCode    String
siteName    String
```

### Database Schema

```sql
CREATE TABLE sites (
    id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id    BIGINT        NOT NULL REFERENCES users(id),
    site_code  VARCHAR(255)  NOT NULL UNIQUE,
    site_name  VARCHAR(50)   NOT NULL
);

CREATE INDEX idx_sites_user_id ON sites(user_id);
```

### Key Design Decisions

**`siteCode` is immutable (`updatable = false`, `unique = true`).** It is used as a URL path variable across every downstream endpoint. If it could be changed, all existing bookmarks and tracking snippet configurations would break instantly. Once assigned, it never changes.

**`siteCode` is system-generated, not user-supplied.** Generating it server-side avoids namespace collision and keeps the creation endpoint simple.

**Site is the ownership anchor.** `siteRepository.findBySiteCodeAndUser(siteCode, user)` is the single ownership check used across campaigns, events, SEO reports, and share logs. One check covers every resource under that site.

## Part 2 — Module 1: Campaign CRUD

Campaigns are marketing efforts scoped to a site. Every campaign belongs to both a `Site` and a `User`.

### API Contract

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/{site_code}/campaigns` | Yes | Create a campaign |
| `GET` | `/api/{site_code}/campaigns` | Yes | List campaigns (paginated, filterable) |
| `GET` | `/api/{site_code}/campaigns/{id}` | Yes | Get a single campaign |
| `PUT` | `/api/{site_code}/campaigns/{id}` | Yes | Update a campaign |
| `DELETE` | `/api/{site_code}/campaigns/{id}` | Yes | Soft-delete a campaign |

**`POST /api/{site_code}/campaigns` — request body:**
```
campaignName           String      required, not blank
campaignDescription    String      optional
cost                   BigDecimal  optional, must be positive if provided
startDate              LocalDate   optional
endDate                LocalDate   optional
status                 Enum        required — ACTIVE | PAUSED | ARCHIVED
channel                Enum        required — ORGANIC | PAID | SOCIAL | EMAIL | EVENT | DIRECT | OTHER
metricName             String      optional (e.g., "Conversions", "Impressions")
metricValue            BigDecimal  optional, must be positive if provided
benchmarkMetricName    String      optional (e.g., "Industry Average Conversions")
benchmarkMetricValue   BigDecimal  optional, must be positive if provided
```

**`GET /api/{site_code}/campaigns` — query parameters:**
```
channel     Enum        optional filter
status      Enum        optional filter
page        int         default 0
size        int         default 20
```

**`PUT /api/{site_code}/campaigns/{id}` — request body:** Same fields as create, all optional. Uses the **null-means-don't-touch** pattern — a `null` field in the request means "leave this field unchanged". To explicitly clear a nullable field, a dedicated endpoint is required.

**`CampaignResponse` — returned by all endpoints:**
```
id                     Long
createdAt              LocalDateTime
campaignName           String
campaignDescription    String      (nullable)
cost                   BigDecimal  (nullable)
startDate              LocalDate   (nullable)
endDate                LocalDate   (nullable)
status                 CampaignStatus
channel                CampaignChannel
metricName             String      (nullable)
metricValue            BigDecimal  (nullable)
benchmarkMetricName    String      (nullable)
benchmarkMetricValue   BigDecimal  (nullable)
isArchived             boolean
```

**Error cases:** `400` validation failure, `404` not found, `403` ownership mismatch.

### Database Schema

```sql
CREATE TABLE campaigns (
    id                     BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id                BIGINT          NOT NULL REFERENCES users(id),
    site_id                BIGINT          NOT NULL REFERENCES sites(id),
    campaign_name          VARCHAR(255)    NOT NULL,
    campaign_description   TEXT,
    cost                   NUMERIC(12, 2),
    start_date             DATE,
    end_date               DATE,
    status                 VARCHAR(20)     NOT NULL DEFAULT 'ACTIVE',
    channel                VARCHAR(20)     NOT NULL,
    is_archived            BOOLEAN         NOT NULL DEFAULT FALSE,
    metric_name            VARCHAR(255),
    metric_value           NUMERIC(12, 2),
    benchmark_metric_name  VARCHAR(255),
    benchmark_metric_value NUMERIC(12, 2),
    created_at             TIMESTAMP WITHOUT TIME ZONE
);

CREATE INDEX idx_campaigns_user_id ON campaigns(user_id);
CREATE INDEX idx_campaigns_site_id ON campaigns(site_id);
CREATE INDEX idx_campaigns_site_archived ON campaigns(site_id, is_archived);
```

### Key Design Decisions

**`NUMERIC(12, 2)` for `cost`, `metricValue`, and `benchmarkMetricValue`.** All three use `BigDecimal` in Java — an exact decimal type. Never use `double` for any value where decimal precision matters.

**`cost` is nullable.** Not every campaign has a tracked spend (e.g., an organic effort may have no direct cost).

**Campaign has both `user_id` and `site_id` FKs.** `user_id` allows direct user-scoped queries without joining through `Site`. `site_id` is the primary organizational anchor. The ownership check uses `site_id` via `checkSiteOwnership`.

**Soft delete via `is_archived`.** Hard-deleting a campaign would orphan all events referencing `campaign_id`. Soft deletes preserve referential integrity and event history.

**All four metric columns are nullable with no default.** Not every campaign has a quantitative goal or benchmark. The null-means-don't-touch pattern in the service means `null` in an update request means "leave unchanged", not "clear it".

## Part 3 — Module 2: Event Tracking (Weeks 4–5)

Events are user interactions captured by a JavaScript tracking snippet. Events are scoped to a `Site` and optionally linked to a `Campaign`.

### API Contract

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/events` | No* | Ingest a single event |
| `GET` | `/api/{site_code}/events` | Yes | Query events with filters + pagination |

*The ingest endpoint is intentionally public — it is called by a JS snippet in a browser with no JWT. Rate limiting is the production mitigation.

**`POST /api/events` — request body:**
```
eventType      String   required — CLICK | PAGE_VIEW | FORM_SUBMIT
pageUrl        String   required, valid URL
siteCode       String   required
campaignId     Long     optional (links event to a specific campaign)
userId         String   optional (anonymous identifier from the client)
utmSource      String   optional
utmMedium      String   optional
utmCampaign    String   optional
channel        Enum     optional — ORGANIC | PAID | SOCIAL | EMAIL | EVENT | DIRECT | OTHER
metadata       Object   optional, freeform JSON
```

**`POST /api/events` — response (201 Created):**
```
id          Long
eventType   String
siteCode    String
country     String    (server-derived from IP, not client-supplied)
createdAt   LocalDateTime
```

**`GET /api/{site_code}/events` — query parameters:**
```
eventType   String    optional filter
startDate   LocalDate optional filter
endDate     LocalDate optional filter
channel     Enum      optional filter
page        int       default 0
size        int       default 50
```

**Error cases:** `400` invalid fields, `404` siteCode not found, `403` ownership mismatch (query only).

### Database Schema

```sql
CREATE TABLE events (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    site_id         BIGINT        NOT NULL REFERENCES sites(id),
    campaign_id     BIGINT        REFERENCES campaigns(id),
    event_type      VARCHAR(50)   NOT NULL,
    page_url        VARCHAR(500)  NOT NULL,
    utm_source      VARCHAR(100),
    utm_medium      VARCHAR(100),
    utm_campaign    VARCHAR(100),
    channel         VARCHAR(50),
    user_identifier VARCHAR(255),
    country         VARCHAR(100),
    metadata        JSONB,
    created_at      TIMESTAMP WITHOUT TIME ZONE
);

CREATE INDEX idx_events_site_created ON events(site_id, created_at DESC);
CREATE INDEX idx_events_site_type ON events(site_id, event_type);
CREATE INDEX idx_events_channel ON events(site_id, channel);
CREATE INDEX idx_events_country ON events(site_id, country);
```

### Key Design Decisions

**Events belong to `Site`, not `Campaign`.** Most events arrive without a campaign context — a direct organic visit has no UTM parameters. `site_id` is always required; `campaign_id` is optional and only populated when UTM parameters indicate a specific campaign.

**Three UTM fields.** `utmSource`, `utmMedium`, and `utmCampaign` enable full attribution: "paid search campaign X via Google" vs. "email newsletter via Mailchimp."

**`channel` as a stored field.** Derived from UTM parameters at ingest and stored explicitly, making analytics queries a simple `GROUP BY channel` rather than runtime string parsing.

**Four composite indexes on `site_id`.** Every query filters by site first. Without these, every event query performs a full table scan across potentially millions of rows.

**Country derived server-side.** Populated using the `geoip2` library against the request IP — never trusted from the client. Server-side geolocation is authoritative (though imperfect for VPN users).

## Part 4 — Module 3: SEO Checker (Weeks 6–7)

The backend crawls a URL, runs a structured HTML audit using **Jsoup**, calls the **Google PageSpeed Insights API**, and persists a structured report scoped to a `Site`.

### API Contract

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/{site_code}/seo/analyze` | Yes | Run an SEO audit on a URL |
| `GET` | `/api/{site_code}/seo/reports` | Yes | Retrieve audit history for a site |

**`POST /api/{site_code}/seo/analyze` — request body:**
```
analyzedUrl   String   required, valid URL
keyword       String   optional (target keyword to check prominence in content)
```

**`POST /api/{site_code}/seo/analyze` — response (201 Created):**
```
id                Long
siteCode          String
analyzedUrl       String
keyword           String          (nullable)
analyzedAt        LocalDateTime
checks            List<SeoCheckResult>
performanceScore  int             (0–100, from PageSpeed)
seoScore          int             (0–100, from PageSpeed)
lcpSeconds        String
fcpSeconds        String
tbtMilliseconds   String
runWarnings       List<String>
```

Where `SeoCheckResult` is:
```
checkName     String       e.g., "TITLE_TAG", "H1_PRESENT"
checkStatus   CheckStatus  PASS | WARN | FAIL
details       String
```

**`GET /api/{site_code}/seo/reports` — query parameters:**
```
analyzedUrl   String    optional filter
page          int       default 0
size          int       default 20
sort          String    default "analyzedAt" (newest first)
```

**Error cases:** `400` invalid URL, `403` ownership mismatch, `503` external scrape failed.

### Database Schema

```sql
CREATE TABLE seo_reports (
    id                BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    site_id           BIGINT        NOT NULL REFERENCES sites(id),
    analyzed_url      VARCHAR(500)  NOT NULL,
    keyword           VARCHAR(255),
    analyzed_at       TIMESTAMP WITHOUT TIME ZONE,
    performance_score INT           NOT NULL DEFAULT 0,
    seo_score         INT           NOT NULL DEFAULT 0,
    lcp_seconds       VARCHAR(50),
    fcp_seconds       VARCHAR(50),
    tbt_milliseconds  VARCHAR(50),
    run_warnings      JSONB,
    pagespeed_raw     JSONB,
    follow_up_at      TIMESTAMP WITHOUT TIME ZONE,
    follow_up_completed  BOOLEAN    NOT NULL DEFAULT FALSE
);

CREATE TABLE seo_checks (
    id             BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    seo_report_id  BIGINT        NOT NULL REFERENCES seo_reports(id),
    check_name     VARCHAR(100)  NOT NULL,
    check_status   VARCHAR(10)   NOT NULL,
    details        TEXT
);

CREATE INDEX idx_seo_reports_site_id ON seo_reports(site_id);
CREATE INDEX idx_seo_reports_follow_up ON seo_reports(follow_up_completed, follow_up_at);
```

### Key Design Decisions

**`SeoReport` belongs to `Site`, not `Campaign`.** The URL being audited is a page on the site. Campaigns don't have their own URLs — they're tagged onto events via UTM parameters.

**`SeoCheckResult` is only for Jsoup HTML checks. PageSpeed metrics are separate columns.** Jsoup checks are pass/warn/fail judgments on specific HTML elements a developer can act on. PageSpeed scores are Google's benchmarks — numeric values, not actionable checks. Keeping them separate means the response surface area is clear and the frontend can display each dimension independently.

**`pagespeedRaw` is stored but excluded from `SeoReportResponse`.** The full PageSpeed JSON can be 100+ KB. Storing it enables future features without re-calling the API. Excluding it from responses keeps payloads lean.

**The `keyword` field is optional.** When provided, `JsoupScraperService` checks keyword prominence in the title, H1, and body. When absent, keyword-related checks are skipped.

**Separate `seo_reports` and `seo_checks` tables, not a JSON blob.** Normalized tables enable cross-report queries: "show me all reports where the title tag check failed." A JSON blob would require application-layer filtering.

**`lcpSeconds`, `fcpSeconds`, `tbtMilliseconds` stored as `VARCHAR`.** PageSpeed returns these as formatted strings with units embedded (e.g., `"2.1 s"`). Storing as strings preserves the original representation with no parsing overhead.

**Two follow-up fields set at creation time.** `follow_up_at` is set to `analyzedAt + 7 days` when the report is first saved. `follow_up_completed` defaults to `false`. Follow-up reports (re-audits triggered by the scheduler) set `follow_up_completed = true` immediately on save, so the scheduler never picks them up again.

**A composite index on `(follow_up_completed, follow_up_at)`.** The daily scheduler query filters on both columns (`follow_up_completed = false AND follow_up_at < now`). Without this index, the scheduler would scan the entire `seo_reports` table on every daily run.

**Graceful failure on external calls.** Both `JsoupScraperService` and `PageSpeedService` wrap all external calls in try-catch. If Jsoup cannot reach the URL, a `503 SeoAnalysisException` is thrown before any database write. If PageSpeed fails, score fields default to `0` and `runWarnings` carries an explanation.

**`SeoAnalysisService.buildAndSaveReport()` is public and reused.** The same method is called for both initial audits (`isFollowUpEmail = false`) and automated follow-up re-audits (`isFollowUpEmail = true`). The boolean flag controls whether `follow_up_at` is set and whether `follow_up_completed` starts as true.

## Part 5 — Module 4: Analytics Dashboard (Weeks 9–10, frontend-led)

A read-only query API that aggregates raw `events` data into summary statistics. All queries are scoped to a `Site`.

### API Contract

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/api/{site_code}/analytics/summary` | Yes | Aggregated event counts |
| `GET` | `/api/{site_code}/analytics/timeseries` | Yes | Daily event counts for charting |

**`GET /api/{site_code}/analytics/summary` — query parameters:**
```
range         String   required — 7d | 30d | 90d
```

**`GET /api/{site_code}/analytics/summary` — response:**
```
siteCode          String
range             String
totalEvents       long
eventBreakdown    Map<String, Long>
channelBreakdown  Map<String, Long>
topCountry        String
topPageUrl        String
```

**`GET /api/{site_code}/analytics/timeseries` — query parameters:**
```
range         String   required — 7d | 30d | 90d
```

**`GET /api/{site_code}/analytics/timeseries` — response:**
```
siteCode      String
dataPoints    List<TimeseriesPoint>
```

Where `TimeseriesPoint` is:
```
date          LocalDate
eventCount    long
```

**Error cases:** `403` ownership mismatch, `404` siteCode not found, `400` invalid range.

### Database Schema

No new tables. Queries aggregate over the `events` table using `@Query` in `AnalyticsRepository`. The composite indexes on `(site_id, created_at DESC)` and `(site_id, channel)` from Module 2 cover the analytics query patterns directly.

### Key Design Decisions

**OLTP-style analytics queries.** Aggregations run directly against the transactional `events` table. The composite indexes make these queries fast at this project's scale. A materialized view would be the next step at high event volumes.

**`range` as a string label.** Accepting `7d`, `30d`, `90d` rather than raw dates simplifies the frontend and makes the API intention-revealing.

**Ownership check still required on read-only endpoints.** An attacker could enumerate competitor analytics by trying sequential site codes. Read-only does not mean access-free.

## Part 6 — Module 5: On-Demand Report Sharing & Automated Follow-Up Audit

This module introduces two features sharing the same infrastructure: emailing a specific SEO report on demand, and an automated 7-day follow-up re-audit that compares new scores against the original. It also introduces Spring Mail for SMTP delivery and `@Scheduled` for time-driven execution.

### The architectural distinction: database-backed polling vs. a simple recurring job

The follow-up audit uses **database-backed polling**. The state lives on the entity (`follow_up_at` and `follow_up_completed` on `SeoReport`). The scheduler is just a daily heartbeat that queries "what's due now?" and processes it. This pattern survives server restarts because the pending work is in the database, not JVM memory — unlike an in-memory `ScheduledExecutorService.schedule()` call that would lose all pending jobs on restart.

**Interview talking point:** *"I used database-backed polling for the follow-up audit. When a report is created, I set `follow_up_at` to 7 days out and `follow_up_completed` to false. A `@Scheduled` job runs daily and queries for reports where `follow_up_at < now` and `follow_up_completed = false`. This pattern survives server restarts because the intent lives in the database."*

### API Contract

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/api/{site_code}/seo/reports/{report_id}/share` | Yes | Email a specific SEO report to any address |
| `GET` | `/api/{site_code}/seo/reports/shares` | Yes | List all share history for the site |

`site_code` and `report_id` come from **path variables**, not the request body. The request body contains only `sentTo`.

**`POST` — request body:**
```
sentTo    String    required, valid email format
```

**`POST` — response (200 OK):**
```
id        Long
sentTo    String
sentAt    LocalDateTime
reportId  Long
```

**`GET /shares` — response (200 OK):**
```
List of:
  id        Long
  sentTo    String
  sentAt    LocalDateTime
  reportId  Long
```

Note: `GET /shares` is scoped to the **site**, not a single report — it returns the full share history for all reports belonging to that site.

**The follow-up audit has no HTTP endpoint** — it is fully automated by the daily scheduler.

### Database Schema

```sql
CREATE TABLE report_logs (
    id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    report_id  BIGINT        NOT NULL REFERENCES seo_reports(id),
    user_id    BIGINT        NOT NULL REFERENCES users(id),
    sent_to    VARCHAR(255)  NOT NULL,
    sent_at    TIMESTAMP WITHOUT TIME ZONE
);

CREATE INDEX idx_report_logs_report_id ON report_logs(report_id);
```

The two follow-up columns on `seo_reports` (see Part 4):
```
follow_up_at         TIMESTAMP WITHOUT TIME ZONE   -- analyzedAt + 7 days on creation
follow_up_completed  BOOLEAN  NOT NULL DEFAULT FALSE
```

### Key Design Decisions

**`ReportSendLog` links to both `SeoReport` and `User`.** `SeoReport` identifies which audit was shared. `User` identifies who triggered the share. Both are needed — `findByReportId()` is used by the follow-up processor to determine the original recipient, and `user` is recorded for audit trail purposes.

**`sentTo` is an explicit column.** The on-demand share can go to any address — a client, colleague, or external stakeholder — not just the account owner. It must be stored explicitly rather than being inferred from the user's email.

**`EmailService` has one generic `sendHtmlEmail()` method.** HTML email bodies are built as strings inside `ReportSendService` via private helper methods: `getReportHtmlEmail()` for the on-demand share email, and `getFollowUpReportHtmlEmail()` which concatenates a delta table (`getChanges()`) with the full new report. This keeps `EmailService` thin — it only knows how to send, not what to say.

**`ReportSendService` owns follow-up orchestration.** The `processFollowUps()` method queries due reports, calls `seoReportService.buildAndSaveReport()` to run the re-audit, builds the comparison email, determines the recipient (first share recipient, or site owner if never shared), sends the email, logs it, and marks the original report `followUpCompleted = true`.

**`SeoAnalysisService` vs. `ReportSendService` — the service split.** `SeoAnalysisService` owns the report lifecycle (scraping, scoring, persisting). `ReportSendService` owns the email lifecycle (sending, logging, follow-up orchestration). Each service has one clear job. The split also allows `buildAndSaveReport()` to be called from both the SEO controller (user-triggered) and the follow-up processor (scheduler-triggered) without coupling them.

**`SeoFollowUpScheduler` is its own class in its own `scheduler/` package.** The scheduler knows *when* to run. The service knows *what* to do. This separation means `ReportSendService.processFollowUps()` can be called directly in tests without waiting for the cron expression.

**Follow-up recipient fallback logic.** The scheduler checks `findByReportId()` to find who the original share recipient was. If the report was never manually shared, it falls back to `report.getSite().getUser().getEmail()` — the site owner. This ensures every follow-up audit reaches someone, even if the on-demand share feature was never used.

**Per-report error isolation.** Each report's follow-up is wrapped in try-catch. If one report fails (PageSpeed down, URL unreachable), the error is logged and processing continues to the next. `follow_up_completed` stays `false` on failure — the scheduler retries it on the next daily run.

**`JOIN FETCH r.checks` in the due-reports query.** The scheduler calls `report.getChecks()` when building the comparison email. Without `JOIN FETCH`, this would trigger N lazy-load queries — one per report — during the scheduler run. `JOIN FETCH` loads all checks in a single SQL JOIN, preventing the N+1 problem.

**`@EnableScheduling` is required on the main application class.** Without it, `@Scheduled` methods are silently ignored — no error, no warning, the job simply never fires. This is one of the most common silent failures in Spring scheduling.
